# GoMarket

## Обзор

GoMarket - интернет-магазин на микросервисной архитектуре: каталог товаров, корзина, оформление заказа с резервированием товара и оплатой с внутреннего кошелька. Проект построен по принципам clean architecture и покрывает ключевые паттерны распределённых систем: синхронную (gRPC) и асинхронную (Kafka) коммуникацию, сагу с компенсациями, transactional outbox, идемпотентность, retry/circuit breaker, наблюдаемость.

Проект является продолжением учебного Cart API: наработанный сервис корзины мигрирует в монорепо и становится одним из микросервисов системы.

**Важно:** все репозитории проекта - приватные.

### Роли

| Роль | Права |
|---|---|
| **Customer** | регистрация, профиль, корзина, заказы, кошелёк, просмотр каталога |
| **Manager** | управление каталогом (товары, категории, остатки), просмотр заказов |
| **Admin** | полный доступ, назначение ролей |
| **Analyst** *(ADV)* | доступ к аналитике |

Роль Customer выдаётся при регистрации; Manager/Analyst назначает Admin.

## Микросервисы

| Сервис | Назначение | Скоуп |
|---|---|---|
| **Gateway** | единая точка входа: маршрутизация, проверка JWT, RBAC, rate limiting | CORE |
| **Auth Service** | выпуск и валидация JWT (access + refresh), сессии | CORE |
| **User Service** | учётные записи, профили, роли, верификация кредов для Auth | CORE |
| **Catalog Service** | товары, категории, остатки, резервирование под заказы | CORE |
| **Cart Service** | корзина пользователя (эволюция Cart API), снапшот для checkout | CORE |
| **Order Service** | оформление заказа, жизненный цикл, оркестрация саги оплаты | CORE |
| **Payment Service** | кошельки, идемпотентные списания/возвраты, outbox | CORE |
| **Notification Service** | уведомления по событиям (заглушки), DLQ | ADV |
| **Search** (в составе Catalog) | полнотекстовый поиск по каталогу на Elasticsearch | ADV |
| **Analytics Service** | сбор событий, статистика продаж | ADV |

## Функциональные требования

### Auth Service

- **Login:** принимает креды, делегирует проверку User Service (gRPC). При успехе выпускает пару токенов.
- **Токены:**
  - access - короткоживущий (15-30 мин), содержит user_id и роли;
  - refresh - долгоживущий (7-30 дней), для обновления пары;
  - активные токены хранятся в Redis с TTL.
- **Валидация:** gRPC-метод для Gateway - проверка подписи, срока и активности в Redis; возвращает user_id и роли.
- **Refresh:** `/refresh` - обмен валидного refresh-токена на новую пару без повторного логина.
- **Logout:** инвалидация обоих токенов в Redis.

### Gateway (NGINX)

- **Маршрутизация:** location/upstream блоки на внутренние сервисы.
- **Аутентификация:** проверка JWT через Auth Service (auth_request); user_id и роли прокидываются внутренним сервисам заголовками - сервисы повторную проверку не делают.
- **RBAC:** ручки Manager/Admin/Analyst закрыты по ролям (401 без токена, 403 при чужой роли).
- **Rate limiting:** limit_req; при превышении - 429.
- **Load balancing:** штатные стратегии NGINX (достаточно конфигурации, демонстрация - ADV).

### User Service

- **Регистрация:** name, email, phone, password; пароль хэшируется (bcrypt/argon2); роль Customer по умолчанию. Публикует `user_registered` в Kafka.
- **Верификация кредов:** gRPC-метод для Auth Service (email/phone + password → user_id, роли).
- **Профиль:** просмотр, обновление (name, phone, email), смена пароля, soft-delete.
- **Роли:** назначение Manager/Analyst админом.

### Catalog Service

- **Товары:** CRUD (Manager): name, description, price, category_id, stock. Публичный просмотр списка и карточки с пагинацией и фильтрами (категория, диапазон цены).
- **Категории:** CRUD (Manager).
- **Остатки:** изменение остатка (Manager); продажа невозможна ниже нуля.
- **Резервирование (внутренние gRPC-методы для Order):**
  - `Reserve(order_id, items[])` - атомарно резервирует все позиции или отклоняет целиком (недостаток остатка → ошибка); идемпотентен по order_id;
  - `ConfirmReservation(order_id)` - списывает резерв в продажу;
  - `ReleaseReservation(order_id)` - возвращает резерв в остаток (компенсация);
  - резерв имеет TTL (~15 мин); фоновый процесс освобождает протухшие резервы.

### Cart Service (эволюция Cart API)

- **Миграция:** код Cart API переносится в монорепо и приводится к общим правилам проекта.
- **Привязка к пользователю:** одна активная корзина на пользователя (user_id из заголовков Gateway); анонимных корзин нет.
- **Позиции:** ссылаются на товары каталога (product_id); при добавлении цена и наличие проверяются в Catalog по gRPC; валидации и лимит позиций из исходного ТЗ сохраняются.
- **Кэш:** чтение корзины через Redis (cache-aside, TTL, инвалидация при изменении).
- **Checkout-поддержка:** gRPC `GetCartSnapshot(user_id)` для Order - фиксирует состав и цены на момент оформления.
- **Очистка:** по событию `order_created` корзина пользователя очищается (идемпотентно).

### Order Service

- **Checkout:** создание заказа из снапшота корзины; пустая корзина → ошибка.
- **Жизненный цикл:** `created → reserved → paid → completed | cancelled` (cancelled - с причиной: нет товара, нет средств, отмена пользователем, таймаут).
- **Сага оформления (оркестрация)** - см. раздел «Сага оформления заказа».
- **Отмена:** пользователь может отменить заказ до статуса paid - запускаются компенсации.
- **История:** список заказов пользователя со статусами; кэш последних заказов в Redis.
- **События:** `order_created`, `order_paid`, `order_cancelled`.

### Payment Service

- **Кошелёк:** создаётся автоматически по событию `user_registered` (идемпотентно).
- **Пополнение:** эндпоинт-заглушка (без реального платёжного провайдера).
- **Списание:** `Charge(order_id, user_id, amount)` - в транзакции БД, идемпотентно по order_id (повторный вызов не списывает дважды); недостаток средств → типизированная ошибка.
- **Возврат:** `Refund(order_id)` - компенсирующая операция, идемпотентна.
- **История:** все операции со статусами; аудит-лог финансовых действий.
- **Outbox:** события `payment_completed` / `payment_failed` записываются в outbox-таблицу в той же транзакции, что и денежная операция; фоновый publisher вычитывает и публикует в Kafka (at-least-once).

### Notification Service (ADV)

- Kafka-consumer: `user_registered`, `order_paid`, `order_cancelled`.
- Уведомления-заглушки (лог/файл/email-мок), шаблоны по типу события.
- Идемпотентная обработка; DLQ для сообщений, не обработанных после N ретраев.

### Search / Elasticsearch (ADV)

- Индексация товаров (по событиям изменения каталога или периодическим синком).
- REST-поиск: полнотекст по name/description + фильтры.
- Graceful degradation: при недоступности ES - фолбэк на Postgres ILIKE, ручка не падает.

### Analytics Service (ADV)

- Kafka-consumer всех бизнес-событий → ClickHouse (допустимо Postgres для упрощения).
- REST-статистика: выручка по дням, топ товаров, конверсия заказов.
- Доступ только с ролью Analyst (через Gateway).

## Сага оформления заказа

Оркестратор - Order Service. Каждый шаг идемпотентен по order_id; состояние саги хранится в БД Order и переживает рестарт сервиса.

```
Checkout
   │
   ▼
[1] Order: создать заказ (created), событие order_created
   │
   ▼
[2] Catalog.Reserve(order_id, items)           ── ошибка ──► cancelled (нет товара)
   │  статус: reserved
   ▼
[3] Payment.Charge(order_id, amount)           ── ошибка ──► компенсация:
   │  статус: paid                                           Catalog.ReleaseReservation
   ▼                                                         → cancelled (нет средств)
[4] Catalog.ConfirmReservation(order_id)
   │  статус: completed, событие order_paid
   ▼
 Done
```

**Компенсации:** ReleaseReservation (возврат резерва), Refund (возврат денег - если сбой случился после успешного Charge). Отмена пользователем до paid запускает те же компенсации.

**Требования к устойчивости:**
- на каждый gRPC-вызов - таймаут;
- retry с exponential backoff + jitter - только на транзиентные ошибки (Unavailable, DeadlineExceeded); бизнес-ошибки (нет средств/товара) не ретраятся;
- circuit breaker на клиентах Catalog и Payment;
- повторное выполнение любого шага (после ретрая или рестарта) не дублирует эффекты;
- незавершённые саги добиваются или компенсируются фоновым процессом (recovery по состоянию из БД).

## Kafka-события

| Событие | Продюсер | Консьюмеры |
|---|---|---|
| `user_registered` | User | Payment (создание кошелька), Notification*, Analytics* |
| `order_created` | Order | Cart (очистка корзины), Analytics* |
| `order_paid` | Order | Notification*, Analytics* |
| `order_cancelled` | Order | Notification*, Analytics* |
| `payment_completed` | Payment (через outbox) | Order, Analytics* |
| `payment_failed` | Payment (через outbox) | Order, Analytics* |

\* - ADV-консьюмеры. Схемы событий описываются в .proto. Все консьюмеры идемпотентны (защита от повторной доставки at-least-once).

Примечание: связка Order↔Payment допускает два варианта - синхронный Charge по gRPC (события payment_* остаются для аналитики/уведомлений) или асинхронный через события. Вариант выбирается со стажёром на этапе BA; синхронный проще и достаточен для CORE.

## Межсервисная коммуникация

**Синхронно (gRPC):** Gateway→Auth (валидация токена), Auth→User (проверка кредов), Cart→Catalog (цены/наличие), Order→Cart (снапшот), Order→Catalog (резервы), Order→Payment (списание/возврат).

**Асинхронно (Kafka):** все события таблицы выше.

**Данные:** database-per-service, прямой доступ к чужой БД запрещён; консистентность между сервисами - событийная (eventual) + сага для оформления заказа.

## Нефункциональные требования

- **Репозиторий:** монорепо `services/*` + `shared/` (proto, общие пакеты).
- **Архитектура сервиса:** обязательные слои handler / service / repository; зависимости через интерфейсы и конструкторы; context пробрасывается всюду.
- **Git:** ветки main (production) и develop; фичи - `feature/TASK-ID-описание` от develop; PR в develop с ID задачи, ревью ментора обязательно.
- **CI/CD (GitHub Actions):** lint (golangci-lint) → unit/integration-тесты (-race) → сборка → Docker-образы (по main). Деплой в k8s - ADV.
- **Тестирование:** unit-тесты бизнес-логики (покрытие ≥ 70-80%), table-driven, моки (mockery/gomock); интеграционные тесты ключевых сценариев; Postman/HTTP-коллекция для e2e.
- **Конфигурация:** env / .env-файлы (viper или envconfig); секреты не коммитятся.
- **Документация:** README на систему и на каждый сервис, Swagger/OpenAPI для REST, .proto для gRPC/Kafka, диаграммы архитектуры и саги (draw.io).
- **Наблюдаемость:**
  - `/metrics` (Prometheus) в каждом Go-сервисе: RED-метрики HTTP/gRPC + бизнес-метрики (заказы/мин, исходы саги, hit-rate кэша);
  - Prometheus + Grafana в docker-compose, общий дашборд, алерты (error rate, недоступность сервиса);
  - structured logging (zap/slog) единого формата с request_id/order_id;
  - distributed tracing (Jaeger) - ADV.
- **Безопасность:** валидация входных данных на границе (handler/Gateway); параметризованные SQL-запросы; bcrypt/argon2 для паролей; rate limiting; аудит финансовых операций.
- **Graceful shutdown** во всех сервисах; liveness/readiness-ручки (`/healthz`, `/readyz`).

## Флоу аутентификации

**Регистрация:** Gateway → User Service → хэширование пароля, сохранение → событие `user_registered` → Payment создаёт кошелёк.

**Login:** клиент → Auth → gRPC-проверка кредов в User → выпуск access+refresh → кэширование в Redis → клиент получает пару.

**Запрос:** клиент → Gateway (access в Authorization) → auth_request в Auth → Auth проверяет подпись/срок/Redis → Gateway получает user_id+роли, проверяет RBAC → проксирует в сервис с заголовками; сервисы токен повторно не проверяют.

**Refresh:** клиент → `/refresh` с refresh-токеном → валидация → новая пара, старая инвалидируется.

**Logout:** оба токена удаляются из Redis и немедленно перестают действовать.

## Критерии MVP (минимум для завершения стажировки)

Сквозной сценарий работает через Gateway и наблюдаем на дашборде:

1. регистрация → автосоздание кошелька → login;
2. Manager наполняет каталог; Customer собирает корзину (с кэшем);
3. пополнение кошелька → checkout → сага: резерв → списание → подтверждение → completed;
4. негативные ветки: нет товара / нет средств → корректные компенсации и cancelled;
5. сбой Payment или Catalog → retry/брейкер отрабатывают, система не зависает, заказ добивается или компенсируется;
6. метрики и алерты отражают происходящее; логи позволяют проследить заказ по order_id.

## Упрощения при нехватке времени (по решению ментора)

- Payment: outbox → прямая публикация после коммита транзакции (списания остаются транзакционными и идемпотентными).
- Оплата целиком: заглушка внутри Order («оплата всегда успешна») - сага сокращается до резерв → подтверждение.
- User на PostgreSQL вместо MongoDB.
- ADV-блок снимается полностью; деплой остаётся в docker-compose.
